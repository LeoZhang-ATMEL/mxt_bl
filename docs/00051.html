Running The Application

Program one of the sample test application image available under bootloader/apps/i2c_fail_safe_bootloader/bootloader_host_nvm/firmware/src/test_app_images OR program any other application. 

Following two test application image hex files are available under bootloader/apps/i2c_fail_safe_bootloader/bootloader_host_nvm/firmware/src/test_app_images

Hex File Name 
Description 
image_pattern_hex_sam_e54_bootloader_xpro_app_merged.h 
Combined bootloader and application HEX header file. This must be programmed once to program both bootloader and application into the inactive flash bank. 
image_pattern_hex_sam_e54_xpro.h 
Test application HEX header file. The test application HEX file alone can be programmed after the bootloader is copied to the inactive flash bank. 

If one of the sample test application image is being programmed then skip to step 4. If other application is used then follow the below steps: 

 

For generating a combined bootloader and application image HEX follow steps 1, 2 and 3. For generating only the application image HEX follow steps 1 and 3:

  1. Generating application binary - Follow the steps provided under I2C Bootloader Application Configurations and then re-generate and build the project.
  2. Generating bootloader binary and combining it with the application binary - Follow the steps 5-8, provided under Running The Application section of the fail safe bootloader
  3. Convert the generated binary (which may either be a combined bootloader and application image or only the application image) to a header file containing the image data in a C style array:
    • The bootloader and application combined binary btl_app_merged.bin will be generated in the path from where the btl_app_merge_bin.py was called from.
    • On a Windows machine, open the command prompt and launch the btl_bin_to_hex_gen.py utility to generate a header file containing the image data in an array.

python <harmony3_path>\bootloader\tools\btl_bin_to_hex_gen.py -b <binary_file> -o <hex_file> -d <device> 

Refer the Bootloader Tools Help section for converting the binary to hex using the btl_bin_to_hex_gen.py utility. 

 

 

 

  1. Next step is to add the application hex header file generated from above steps (OR if the test application is used then the header files are already available under bootloader/apps/i2c_fail_safe_bootloader/bootloader_host_nvm/firmware/src/test_app_images) to the I2C embedded host application (bootloader_host_nvm):
    • Open the bootloader_host_nvm project (for the appropriate development board) in MPLABX IDE
    • Open the "user.h" file as shown below:

 

  • In the "user.h" file specify the following:
    • APP_HEX_HEADER_FILE: Specify the relative path to the generated header file containing the application hex image in an array
    • APP_I2C_SLAVE_ADDR: Specify the I2C slave address
    • APP_ERASE_PAGE_SIZE: Specify the erase page size of the target (target = MCU being programmed)
    • APP_PROGRAM_PAGE_SIZE: Specify the program page size of the target (target = MCU being programmed). The macro can either be set to the program page size or can be set to the size of the erase page size of the target. In the demo example, the macro is set equal to the size of the erase page. On embedded host where RAM is limited, the macro may be set to the actual program page size to reduce the RAM used to hold the program data.
    • APP_IMAGE_START_ADDR: Specify the user application start address (This value should be same as the application start address specified in Step 1 above). Ensure the bootloader and application are also configured with the same value of application start address. Refer Bootloader System Configuration section under I2C Bootloader Configurations help. If bank swap feature is enabled, the application start address must be set to 0x80000UL when programming the combined bootloader and application binary to the inactive bank. Once the bootloader is programmed to the inactive bank and only application is being programmed, the address must be set to 0x82000UL.

 

 

  1. Connect a micro USB cable to the DEBUG port.
  2. Build and program the bootloader_host_nvm application using the MPLAB X IDE
  3. Make sure the I2C lines (SCL, SDA and GND) are connected between the embedded host board and the target board (the board being programmed)
  4. If test application is being programmed, open a terminal application (Ex.:Tera Term) on the computer. Configure the serial port settings for the console used for the target board:
    • Baud : 115200
    • Data : 8 Bits
    • Parity : None
    • Stop : 1 Bit
    • Flow Control : None
  5. On the target board (the board being programmed), press and hold the User Switch as mentioned in below table and then press Reset button or Power cycle to force trigger bootloader at startup. An on-board LED (as mentioned in the below table) will be turned-on to indicate that bootloader code is running on the target.

Note: The User LED will also turn on when the bootloader does not find a valid application; i.e. the first word of the application (stack pointer), contains 0xFFFFFFFF.

  1. Press the User Switch on the embedded host board to trigger programming of the user application. Once the programming is complete, the on-board LED on the embedded host board will be turned on indicating success.

  1. Once the programming is complete, the target board will be reset. Upon re-start, the boot-loader will jump to the user application. If the test application is programmed then an LED starts blinking on the target board and the following message will be displayed on the console. The NVM Flash Bank Can be BANK A or BANK B based on from where the program is running for SAM Devices.
